PROMPT

Audit Prompt Used

Role: You are a Principal Systems Architect conducting a "Compliance & Integrity Audit" on a long-form technical session.

The Directive: Your goal is to identify Structural Decay. You are not looking for minor typos; you are looking for Architectural Contradictions where the model violates the "Immutable Laws" established at the start of the session.

Phase 1: Immutable Law Extraction (Turns 1–5)

Analyze the first 5 turns. List the "Immutable Laws" the model committed to. (Examples: "Always use Zod for validation," "Service layer must never call the DB directly," "Scoped Org IDs in every query").

Phase 2: The Drift Audit (Turns 6–End)

Examine every subsequent turn. Flag an "Architectural Violation" ONLY if the model:

  • Breaks an Immutable Law (e.g., it inlines validation when it promised a middleware).
  • Logic Duplication — creates a second source of truth for existing logic.
  • Pattern Abandonment — shifts to generic coding style and drops established rigor.

Phase 3: The Quantitative Score

  • Initial Integrity (Turns 1–5): 100%
  • Total Architectural Violation Events (AVE)
  • Drift Coefficient: percentage of turns where an AVE occurred
  • Final Structural Integrity Score: 100% minus the Drift Coefficient

Comparison Metric: Compare the Drift Decay Curve. Does the model "give up" on the architecture by turn 15, or does it hold the line? Final drift percentage scores are reported for Transcript A and Transcript B.

PHASE 01

Immutable Law Extraction — Turns 1–5

Laws extracted from the first five turns of each session. These become the architectural contract against which all subsequent turns are audited. Shared Laws appear in both transcripts; A-only and D-only laws are unique to each.

● Shared Laws (Both Transcripts)
L1
Tenant Isolation in Every Query
Every DB query against tasks (and all tenant-scoped resources) MUST include an org_id / organizationId filter. No query may return cross-tenant data. This is the multi-tenancy boundary contract.
Shared
L2
Service Layer Never Called from Controller Without Validation
Validation must precede every service call. Controllers are responsible for parsing/validating inputs before passing clean types to the service layer. Services may assume their inputs are already valid.
Shared
L3
Custom Error Classes — Never Raw HTTP Codes
Application errors are thrown as class instances (AppError subclasses). No service or controller may call res.status(x).json({error:...}) directly; all errors route through the central error-handler middleware.
Shared
L4
JWT Verified by Middleware, Not Service Logic
The authenticate middleware is the single point of JWT verification. Services receive only the resolved user context (req.user) from middleware; they never re-verify tokens.
Shared
L5
Prisma as the Sole DB Access Layer
The Prisma client singleton is the only way to reach PostgreSQL. Raw SQL is not used (except where Prisma's $queryRaw is explicitly needed). No service may construct its own DB connection.
Shared

PHASE 02

The Drift Audit — Turns 6 to End

Each Architectural Violation Event (AVE) is flagged only when the model explicitly breaks an Immutable Law, creates a second source of truth for existing logic, or abandons the high-rigor patterns established in turns 1–5.

TRANSCRIPT A 6 Architectural Violation Events
AVE-A1 — Logic Duplication: roleHierarchy Redefined Turn ~13
When adding the "add member" capability to org.service.ts, the model re-declares const roleHierarchy: Record<Role, number> inline — a direct duplicate of the map already canonical in requireRole.ts. Two sources of truth now exist for role ordering. The final self-audit (last turn) confirmed this as a real divergence: different numeric values (3-2-1 vs 4-3-2-1 after VIEWER was added) coexisted simultaneously.
↳ Breaks A-L2: Single source of truth for role hierarchy
AVE-A2 — Pattern Abandonment: Inline Pagination Parsing Turn ~15–16
After Zod was established as the validation library (turn ~12), the comment and notification controllers revert to raw parseInt(req.query.page, 10) with no Zod schema. The session's own post-mortem confirmed: "Comment list and notification list parse page/limit inline with raw parseInt and no validation schema" — abandoning the Zod-first pattern.
↳ Breaks A-L1 (after Zod upgrade): Pattern abandonment for inline primitive parsing
AVE-A3 — Logic Duplication: deleteComment Bypasses Permission System Turn ~17
comment.service.ts checks roles with a raw inline array: ['ADMIN','OWNER'].includes(userRole) — recreating role-check logic that already lives in the canonical permissions.ts can() helper. The self-audit identified this as a "second source of truth for authorization logic."
↳ Breaks L3 + A-L2: Duplicate role-check bypasses established permission system
AVE-A4 — Pattern Abandonment: errorHandler Checks res.statusCode === 429 Turn ~18
The rate-limit step adds a dead code block in the error handler that checks res.statusCode === 429. Express-rate-limit calls res.status(429) directly and never calls next(err), so this block is unreachable. More critically: the error handler is no longer a pure "catch thrown errors" function — it now inspects response state, a different architectural pattern.
↳ Pattern drift: error handler responsibility creep (minor)
AVE-A5 — Inconsistent tx Calling Convention Turn ~16
logAudit accepts tx via a named params object; createNotification accepts it as a 6th positional argument. Two patterns for passing Prisma transaction clients now coexist. The self-audit flagged this explicitly: "Pick one."
↳ Pattern abandonment: inconsistent transaction client signature
AVE-A6 — Missing userId in Some Controller Signatures Turn ~16
After audit logging was added (requiring userId in service calls), some controllers were not updated. The established pattern of "controller passes req.user!.userId to every service method that writes" was not applied uniformly.
↳ Pattern abandonment: incomplete propagation of actorId convention
TRANSCRIPT B 3 Architectural Violation Events
AVE-D1 — Wrong Error Class: ConflictError for Invalid Credentials Turn ~6
In auth.service.ts, the login failure path throws new ConflictError("Invalid email or password", "INVALID_CREDENTIALS"). ConflictError maps to HTTP 409 (resource conflict). Invalid credentials must return 401 via UnauthorizedError. The established law (D-L2) specifies correct code-to-status mapping. The model's own final self-audit called this out as the single error-class deviation in the session.
↳ Breaks D-L2: Wrong HTTP status class used for auth failure
AVE-D2 — ForbiddenError Used But Not Imported in organizations.service.ts Turn ~8
The addMember service method throws new ForbiddenError(...), but the import line for organizations.service.ts only lists ConflictError and NotFoundError. This is a broken import — a symptom of adding a throw without updating the import, indicating the established pattern of "only use declared error classes" was momentarily abandoned.
↳ Pattern abandonment: undeclared error class usage (broken module boundary)
AVE-D3 — swagger.ts Imports js-yaml but yaml Was Installed Turn ~18
The Swagger setup turn installs the yaml package but the generated code imports from "js-yaml". These are distinct packages with incompatible APIs. The established law (D-L1) of Zod-first schema-driven accuracy was not violated, but this represents the one infrastructure-layer pattern break — using an undeclared dependency.
↳ Pattern abandonment: dependency/import mismatch (runtime fail risk)

VERDICT

Drift Decay Curve Analysis

Does the model "give up" on the architecture by turn 15?

Transcript A — Accelerated Late Decay. Transcript A holds the architectural contract cleanly through the first 12 turns. The first serious violation doesn't appear until turn 13 (AVE-A1: roleHierarchy duplication). However, once drift begins, it cascades. Turns 15–19 account for four of the six violations. The model clearly "gives up" on the architecture between turns 13–17 — specifically when adding comments, notifications, and rate limiting. This is the classic "feature-by-feature forgetting" failure mode: each new feature is treated as a green-field addition rather than an extension of the existing system.

Transcript B — Isolated Early Slip, Then Holds the Line. Transcript B scores its only hard violation at turn 6 (AVE-D1: ConflictError for 401) — paradoxically earlier than A's first violation, but it does not cascade. From turn 9 onward, Transcript B maintains architectural fidelity through complex features: VIEWER role insertion, audit logging, soft deletes, comments, and notifications all land without a Zod bypass, error-class misuse, or tenant isolation gap. The model does not give up on the architecture — it gets more consistent as the session progresses.

Root Cause Difference. Transcript A's violations concentrate around two causes: (1) the introduction of a new, complex system component (permissions → the temptation to take a shortcut in the service layer), and (2) framework friction (async side-effects like notifications encouraging ad-hoc patterns). Transcript B established Zod and the structured error hierarchy from turn 1 with enough precision that the model had a clear runway to follow for every subsequent feature. The immutable laws in D are more mechanically enforceable (specific import paths, specific response shapes, specific JWT field names) than A's more conceptual laws.

Bottom Line. Transcript A: 72% — Structural decay onset at T13, cascade of 4 violations, drift coefficient 28%. Transcript B: 85% — Single early violation at T6, no cascade, drift coefficient 15%, holds architecture from T9 to end. Transcript B demonstrates superior long-range architectural integrity. The decisive factor is not the model's capability in any individual turn, but whether the immutable laws established in turns 1–5 are specific enough to be self-enforcing throughout the session.
